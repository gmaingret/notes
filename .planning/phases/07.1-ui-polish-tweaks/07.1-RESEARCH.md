## Summary

Phase 7.1 is four surgical edits to existing React components. No new routes, no new hooks beyond wiring `onSuccess` on `useCreateDocument`, no new dependencies. All work is component-level inline style tweaks, Lucide import swaps, and a small prop threading exercise.

The trickiest piece is the `+ button` flow: `useCreateDocument` must add an `onSuccess` callback to capture the new document's ID, then pass that ID as `pendingRenameId` down to `DocumentList`, which passes it to the matching `DocumentRow`. `DocumentRow` already has a fully-functional `isRenaming`/`renameInputRef` pattern — the only new wiring is a way to *externally* trigger that rename state from the parent rather than from the in-row `...` menu.

The FocusToolbar icon swap is straightforward mechanical work: nine HTML entities replaced with nine Lucide imports, all using the established `size={20}` / `strokeWidth={1.5}` standard. The bookmark and complete/delete icons are NOT in the locked decision table — they need special attention (see Pitfall 2 below).

**Primary recommendation:** Implement in four independent tasks (one per tweak), in order: spacing, FocusToolbar icons, sidebar footer, + button flow. Each task is independently testable and carries no cross-task dependencies.

### Pattern 4: + Button with Pending Rename (the complex one)
**What:** Three-step sequence on `+` click — tab switch, document create, trigger rename. Implemented via `pendingRenameId` state in `Sidebar`, passed as prop to `DocumentList`, passed as prop to `DocumentRow`.

**Key insight on `useCreateDocument` mutation:** The current `onSuccess` in the hook itself only calls `invalidateQueries`. The `mutate` call in `Sidebar.tsx` needs an inline `onSuccess` to capture the new doc ID:

```typescript
// In Sidebar.tsx
const [pendingRenameId, setPendingRenameId] = useState<string | null>(null);
const { mutate: createDocument } = useCreateDocument();

const handleCreate = () => {
  if (sidebarTab !== 'docs') setSidebarTab('docs');
  createDocument(undefined, {
    onSuccess: (newDoc) => {
      setPendingRenameId(newDoc.id);
    },
  });
};
```

The `@tanstack/react-query` v5 `mutate` function accepts a second argument `{ onSuccess, onError, onSettled }` — this is documented behavior. The hook-level `onSuccess` (invalidateQueries) fires first, then the call-site `onSuccess` fires. This means React Query will have already re-fetched the documents list by the time `pendingRenameId` is set, so the new doc row will be in the DOM.

**Pass pendingRenameId down the tree:**

```typescript
// Sidebar.tsx — DocumentList usage
<DocumentList activeDocId={activeDocId} pendingRenameId={pendingRenameId} onRenameComplete={() => setPendingRenameId(null)} />
```

```typescript
// DocumentList.tsx — updated Props type
type Props = {
  activeDocId: string | null;
  pendingRenameId?: string | null;
  onRenameComplete?: () => void;
};
```

```typescript
// DocumentList.tsx — pass to DocumentRow
<DocumentRow
  key={doc.id}
  document={doc}
  isActive={doc.id === activeDocId}
  initiallyRenaming={doc.id === pendingRenameId}
  onRenameComplete={doc.id === pendingRenameId ? onRenameComplete : undefined}
/>
```

**DocumentRow receives `initiallyRenaming` prop** — triggers `isRenaming` state on mount:

```typescript
// DocumentRow.tsx — updated Props
type Props = {
  document: Document;
  isActive: boolean;
  initiallyRenaming?: boolean;
  onRenameComplete?: () => void;
};

// In DocumentRow body
const [isRenaming, setIsRenaming] = useState(initiallyRenaming ?? false);

// Modify handleRenameSubmit to call onRenameComplete
const handleRenameSubmit = () => {
  const trimmed = renameValue.trim();
  if (trimmed && trimmed !== document.title) {
    rename({ id: document.id, title: trimmed });
  }
  setIsRenaming(false);
  onRenameComplete?.();
};
```

**Timing note:** `invalidateQueries` triggers a re-render of DocumentList with the new doc. At that point `pendingRenameId` is set, so the new `DocumentRow` is mounted with `initiallyRenaming={true}`, which seeds `useState(true)` and auto-focuses the input. No `useEffect` needed — `useState(initiallyRenaming ?? false)` handles it at mount time.

### Anti-Patterns to Avoid
- **Setting `pendingRenameId` before `createDocument` fires:** The new doc ID is only available in `onSuccess`. Never try to anticipate it with a timestamp or temp ID.
- **Using `useEffect` to drive rename focus:** `useState(initiallyRenaming)` seeded at mount is simpler and avoids double-render timing issues.
- **Modifying the `useCreateDocument` hook's own `onSuccess`:** The hook-level `onSuccess` handles cache invalidation — don't overload it. Use the call-site `onSuccess` argument instead.
- **Rendering `<input>` inside dnd-kit `useSortable` listeners:** `DocumentRow` already has `{...listeners}` spread on the outer div. The rename `<input>` already calls `e.stopPropagation()` on `onClick` — this is correct and already implemented in the existing rename flow.
- **Applying padding to the inner translated div:** Bullet spacing belongs on the outer div (dnd-kit ref). Applying it to the inner swipe-translated div would cause the swipe background reveal to not cover the padding area.

## Common Pitfalls

### Pitfall 1: FocusToolbar — Bookmark and Complete/Delete Icons Not in Locked Mapping
**What goes wrong:** The locked decision table only covers 8 icons. `FocusToolbar.tsx` has 11 buttons total. The bookmark (&#128278;), complete (&#10003;), and delete (&#128465;) buttons also use Unicode chars and are NOT in the mapping table.
**Why it happens:** The discussion focused on the motion-related icons; status icons were implicitly included but not listed.
**How to avoid:** VISL-01 already requires no Unicode chars in any component. Use: `<Bookmark />` (fill toggled via class), `<Check />`, `<Trash2 />` for the remaining three. Match the existing `BulletNode.tsx` precedent which already uses `<Check>` and `<Trash2>` for swipe icons.
**Warning signs:** Test assertion `not.toContain('&#128278;')` would catch this; add it.

### Pitfall 2: `onSuccess` Call-Site vs Hook-Level Firing Order
**What goes wrong:** The call-site `onSuccess` in `mutate(undefined, { onSuccess })` fires AFTER the hook's `onSuccess` (which calls `invalidateQueries`). However, `invalidateQueries` is async — the query re-fetch may not have completed by the time call-site `onSuccess` runs.
**Why it happens:** `invalidateQueries` marks queries stale and triggers background re-fetch; it does not `await` the fetch. The new doc is in the response of the `mutationFn` itself, so `newDoc.id` from call-site `onSuccess` is reliable immediately.
**How to avoid:** Use the `newDoc` returned directly by the `mutationFn` (it is the POST response body — a `Document` object). Don't wait for the refetch to determine the ID.
**Warning signs:** If `pendingRenameId` is set but the row isn't rendered yet, it means the query refetch is still in flight. The row will appear on the next render after the refetch completes, and `pendingRenameId` will still be set — the rename input will mount correctly on that render.

### Pitfall 3: `...` Button Removal Leaves `showSidebarMenu` State Orphaned
**What goes wrong:** Removing the `...` button without removing the `showSidebarMenu` useState and the associated overlay/dropdown JSX causes a TypeScript "variable declared but never read" warning (or just dead code).
**How to avoid:** Remove the entire `showSidebarMenu` state, the `setShowSidebarMenu` calls, the backdrop `<div>`, the `dropdownStyle` const, and the `dropdownItemBase` const together. Verify `exportAll` and `handleLogout` are still called from the new footer.

### Pitfall 4: Sidebar Footer Hover Style Must Use CSS Class (Not Inline Style)
**What goes wrong:** Using inline `onMouseEnter`/`onMouseLeave` to toggle background color breaks in dark mode — the value would be a hardcoded color, not a CSS token.
**Why it happens:** Phase 6 already established this anti-pattern and fixed it. The ContextMenu hover was migrated from JS to `.context-menu-item:hover` CSS rule.
**How to avoid:** Add `.sidebar-footer-btn:hover { background: var(--color-bg-surface); }` to `index.css`. Keep the button `color` as inline style using the token var (or use a CSS class).

### Pitfall 5: `DocumentRow` `useSortable` + `initiallyRenaming` Interaction
**What goes wrong:** When `DocumentRow` is mounted with `initiallyRenaming={true}`, the `{...listeners}` from `useSortable` are spread on the outer div. The rename `<input>` with `autoFocus` causes the outer div to not receive pointer events properly during focus. The dnd-kit listeners would intercept touch/pointer events on the input.
**How to avoid:** The existing pattern already handles this — the rename `<input>` calls `e.stopPropagation()` on `onClick`. Also, dnd-kit's `PointerSensor` uses `activationConstraint: { distance: 5 }` so a simple click/focus won't activate drag. No change needed — just verify the existing `stopPropagation` is on the `<input>` element.
