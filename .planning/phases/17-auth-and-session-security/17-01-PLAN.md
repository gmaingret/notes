    - validatePasswordPolicy("alllowercase1") returns error: must include at least one uppercase letter
    - validatePasswordPolicy("ALLUPPERCASE1") returns error: must include at least one lowercase letter
    - validatePasswordPolicy("NoDigitsHere!") returns error: must include at least one digit

    - validatePasswordPolicy("V4l!dPass") returns null (valid)
    - validatePasswordPolicy("C0mpl3x!Pass") returns null (valid)
  </behavior>
  <action>
    Create `server/src/services/passwordPolicy.ts` exporting a `validatePasswordPolicy(password: string): string | null` function that returns an error message string if invalid, or null if valid.

    Character diversity rules (3-of-4 is overcomplicated for a self-hosted app; enforce all three essentials):
    - At least one uppercase letter
    - At least one lowercase letter
    - At least one digit
    - Minimum 8 characters (defense-in-depth; Zod also checks)

    Common password check: Embed a static array of ~100 commonly breached passwords (e.g., "password", "Password1", "123456", "qwerty123", "letmein", "welcome1", etc.). Check case-insensitively. This is simpler and more reliable than an npm package for a self-hosted single-user app.

    Return the FIRST failing rule as a specific, user-friendly error message, e.g.:
    - "Password must be at least 8 characters"
    - "Password must include at least one uppercase letter"
    - "Password must include at least one lowercase letter"
    - "Password must include at least one digit"
    - "This password is too common. Please choose a more unique password."

    Create test file `server/tests/passwordPolicy.test.ts` with the behaviors listed above. Run RED first, then GREEN.
  </action>

    <automated>cd server && npx vitest run tests/passwordPolicy.test.ts</automated>
  </verify>
  <done>validatePasswordPolicy rejects weak/common passwords with specific messages, accepts strong passwords</done>

       - Line 165: Change `/?token=${encodeURIComponent(accessToken)}` to `/#token=${encodeURIComponent(accessToken)}`
       - This single character change (? to #) ensures the token is in the hash fragment, never sent to the server in subsequent requests
       - SESS-02 is already satisfied: client AuthContext.tsx reads from window.location.hash and clears via replaceState
