<objective>
Add automatic silent token refresh to the web client so expired access tokens are refreshed transparently. When a request gets a 401, the ApiClient calls a refresh handler (injected by AuthContext), retries the original request with the new token, and uses a shared promise lock to prevent concurrent refresh calls. A retry guard (_isRetry flag) prevents infinite loops. If refresh fails, the user is logged out with a toast notification.

Purpose: Users currently get silently logged out when their 15-minute access token expires mid-session. This caused a real bug during Phase 20 UAT.
Output: Modified client.ts with 401 interceptor, modified AuthContext.tsx with handler injection.
</objective>

   - `private refreshPromise: Promise<string | null> | null = null`

3. Add a private helper method `handleUnauthorized` that encapsulates the 401 retry logic:
   ```typescript
   private async handleUnauthorized<T>(
     retryFn: (token: string) => Promise<T>
   ): Promise<T> {
     if (!this.refreshHandler) throw new Error('Session expired');

     // Shared promise lock: reuse in-flight refresh
     if (!this.refreshPromise) {
       this.refreshPromise = this.refreshHandler().finally(() => {
         this.refreshPromise = null;
       });
     }

     const newToken = await this.refreshPromise;

     if (!newToken) {
       // Refresh failed — session truly expired
       if (this.logoutHandler) this.logoutHandler();
       throw Object.assign(new Error('Session expired'), { status: 401 });
     }

     this.setToken(newToken);
     return retryFn(newToken);
   }
   ```

4. Modify `request()` method: After the `fetch()` call, BEFORE the existing `if (!res.ok)` block, add 401 interception:
   - Check `if (res.status === 401 && !(options as any)?._isRetry)`
   - If true, call `return this.handleUnauthorized((token) => this.request(path, { ...options, headers: { ...options.headers }, _isRetry: true } as any))`
   - The `_isRetry` flag on the retry call prevents infinite loops — if the retried request also returns 401, it falls through to the normal error throw

5. Modify `upload()` method: After the `fetch()` call, BEFORE the existing `if (!res.ok)` block, add the same 401 check:
   - Check `if (res.status === 401 && !(options as any)?._isRetry)` — but upload doesn't take options, so use a parameter or internal flag
   - Simplest approach: add an optional `isRetry?: boolean` parameter to `upload()`:
     `async upload<T>(path: string, formData: FormData, isRetry = false): Promise<T>`
   - On 401 when !isRetry: `return this.handleUnauthorized((token) => this.upload(path, formData, true))`
   - The retry call in handleUnauthorized already calls `this.setToken(newToken)` so the retried upload() will pick up the new token from `this.accessToken`

6. Modify `download()` method: Same pattern as upload:
   - Add `isRetry = false` parameter: `async download(path: string, isRetry = false): Promise<Response>`
   - After fetch, check `if (res.status === 401 && !isRetry)`
   - On 401: `return this.handleUnauthorized(() => this.download(path, true))`
